# 质量保障规范

## 质量保障的目标

- 质量保障的目标是确保系统不仅“能运行”，而且行为可靠、契约稳定、结果可解释、交付可维护。
- 质量保障应覆盖实现、测试、文档、评审和交付全过程，而不应只依赖单一环节。
- 任何重要功能的交付都应同时关注正确性、稳定性、可维护性和可追踪性。
- 质量保障不是额外负担，而是降低返工成本和长期维护成本的必要机制。

## 质量保障的基本原则

- 质量问题应尽量在开发阶段暴露，而不是推迟到运行阶段或交付之后。
- 越靠近公共边界、核心流程和稳定契约的改动，越应提高质量要求。
- 任何会影响他人使用、系统行为或外部契约的改动，都应同步进行验证和说明。
- 若某项实现难以解释、难以验证、难以维护，通常意味着质量仍不达标。

## Code Review 规范

- 所有重要改动都应接受 Code Review，重点关注行为风险，而不仅是代码风格。
- Review 应优先检查：
  - 模块职责是否清晰
  - 分层是否被破坏
  - 是否引入了不必要的耦合
  - 输入输出契约是否明确
  - 是否影响现有公共行为
  - 是否缺少必要测试
  - 是否存在文档与实现不一致
- Review 的目标是发现真实问题、降低后续风险，而不是形式化通过流程。
- 若某项改动无法被 reviewer 清楚解释其边界和影响范围，应视为需要进一步澄清。

## 文档一致性规范

- 任何影响公共行为、目录结构、文件契约、模块职责或运行方式的改动，都应同步更新文档。
- 文档应视为系统契约的一部分，而不是可有可无的附属材料。
- 如果实现已经改变而文档未更新，应视为质量缺陷，而不是单纯文档遗漏。
- 新增公共模块、公共约定或稳定接口时，应补充最基本的说明文档。

## 契约稳定性规范

- 任何被多个模块、多个流程或外部使用方依赖的输入输出约定，都应视为稳定契约。
- 稳定契约可以表现为数据结构、接口形式、配置约定、文件格式、目录组织、调用方式或其他外部可见行为。
- 修改稳定契约时，应明确：
  - 改动内容
  - 影响范围
  - 是否兼容旧行为
  - 是否需要同步调整依赖方
- 对外部可见的行为，不应在缺乏说明和验证的情况下静默改变。
- 契约一旦形成，就应以可演进、可验证、可追踪的方式进行管理。

## 变更影响控制规范

- 在提交重要改动前，应主动评估其影响范围，而不是只关注当前局部是否可运行。
- 对公共能力、公共接口或核心流程的改动，应特别关注其是否会影响：
  - 其他模块
  - 其他流程
  - 既有契约
  - 测试预期
  - 运行入口
  - 外部使用方式
- 若改动跨越多个层次或多个边界，应优先拆分，避免把多种变化混在同一提交中。
- 若存在高风险但短期无法彻底解决的问题，应明确记录其现状、影响和限制条件，而不是隐含保留。

## 完成定义规范

- 一个功能只有在以下条件同时满足时，才可认为基本完成：
  - 实现完成
  - 相关测试通过
  - 关键行为得到验证
  - 文档已同步更新
  - 无明显调试残留
  - 无已知未说明的严重风险
- “代码已写完”不等于“工作已完成”。
- 对核心流程和公共能力，完成标准应高于普通局部修改。

## 缺陷处理规范

- 已发现的缺陷应优先保证问题可复现、可定位、可解释。
- 修复缺陷时，应优先补充对应的回归保护，而不是只做局部修补。
- 若缺陷暂时无法彻底解决，应明确记录现状、影响和临时约束。
- 不应通过隐藏错误、吞掉异常或模糊行为来制造“看起来没问题”的假象。

## 交付质量检查规范

- 在准备提交、合并或交付前，应进行最基本的质量检查。
- 至少应确认：
  - 代码状态干净
  - 相关测试通过
  - 文档与实现一致
  - 配置和脚本没有明显漂移
  - 没有临时调试输出、硬编码路径或失效注释
- 对涉及核心功能或公共契约的改动，应提高检查强度，而不是沿用最低标准。
- 交付前的检查不是形式流程，而是防止低级错误进入主线的最后一道保障。

## 长期可维护性规范

- 质量保障不仅面向当前提交，也面向未来维护者。
- 一个改动如果只有作者本人能解释、能运行、能修复，则说明质量仍不够高。
- 设计、实现、测试、文档应共同支持后续重构、扩展和问题排查。
- 长期质量的核心不是“从不出错”，而是“出错后能快速发现、解释、修复，并避免重复发生”。
