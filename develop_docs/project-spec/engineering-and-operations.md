# 工程化与运维规范

## 日志规范

- 运行中的关键信息应通过统一日志机制记录，不应依赖随意的 `print` 输出。
- 日志应服务于流程追踪、问题定位和运行分析，避免输出无意义或重复信息。
- 错误日志应尽量包含足够的上下文，以便定位失败发生的阶段、对象和原因。
- 调试日志可以存在，但应可控、可关闭，不应长期污染正常运行输出。

## 配置管理规范

- 配置应集中管理，不应在代码中散落硬编码参数。
- 不同层级的配置应有清晰归属，例如环境级配置、运行级配置、任务级配置应分开管理。
- 配置的来源、优先级和覆盖关系应明确，避免多个入口同时控制同一行为却无统一规则。
- 对外部依赖、模型参数、路径和运行开关等关键配置，应提供稳定、可追踪的配置入口。

## 环境与依赖规范

- 运行环境应尽量可复现，避免依赖“作者本机默认状态”才能运行。
- 外部依赖应显式声明，避免隐式依赖系统环境中的偶然安装结果。
- 涉及系统工具、模型、代理、硬件资源等运行前置条件时，应明确说明检查方式和准备步骤。
- 新增依赖前，应先评估其必要性、兼容性和维护成本。

## 路径与文件系统规范

- 路径处理应保持稳定、清晰、可移植，避免硬编码个人机器路径或临时目录约定。
- 涉及文件写入、目录创建、资源复制和中间产物管理时，应统一规划存放位置。
- 临时文件、缓存文件、生成文件、持久化结果文件应在语义上分离，避免互相污染。
- 所有涉及文件系统的行为都应假设路径可能不存在、不可写或状态异常，并进行必要保护。

## 资源管理规范

- 涉及大文件、长任务、视频处理、模型调用等资源密集型操作时，应显式考虑资源占用和生命周期。
- 不应无控制地重复复制大文件、重复抽取资源或无限保留中间产物。
- 中间结果如果需要保留，应有明确目的；如果不需要保留，应及时清理或集中到约定目录。
- 对 CPU、内存、磁盘、GPU、网络请求等关键资源的使用，应优先保证稳定性和可恢复性。

## 异常处理规范

- 异常处理应优先保证问题可见、原因可追踪，而不是简单吞掉异常。
- 不同层应处理自己职责范围内的异常，避免所有异常都在最底层或最高层统一模糊处理。
- 对可预期失败应尽量提供清晰错误信息，对不可预期失败应保留足够上下文供后续排查。
- 不应为了“流程继续跑完”而忽略已经破坏核心契约的错误。

## 运行入口与脚本规范

- CLI、脚本和配置文件的职责应清晰，避免多个入口承担重复或冲突的配置职责。
- 脚本应尽量保持单一用途，输入输出明确，可直接用于验证或批处理。
- 运行入口应优先服务于可重复执行和可调试，而不是只为一次性本地操作方便。
- 临时实验脚本若逐渐成为稳定工作流的一部分，应及时整理为正式入口并补齐文档。

## 可观测性与可维护性规范

- 一个系统不仅要能运行，还要能被观察、调试、解释和维护。
- 对关键流程应能回答：它现在在做什么、失败在哪里、输出了什么、依赖了什么。
- 任何运行时行为如果难以观察、难以解释、难以复现，通常意味着工程化设计仍不完善。
- 工程化设计的目标不是增加流程复杂度，而是让系统在真实环境中更稳定、更透明、更容易接手。
