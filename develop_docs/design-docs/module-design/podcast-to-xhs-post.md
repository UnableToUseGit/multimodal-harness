# Podcast To XHS Post

## 模块目标

`podcast-to-xhs-post` 是一个面向 agent 的内容创作 skill。

它的职责不是直接理解原始播客，而是：

1. 先借助 `mm-harness` 将播客或视频播客转换为结构化 workspace
2. 再由 agent 从该 workspace 中提炼信息
3. 生成适合小红书发布的笔记成品，包括：
   - 标题
   - 正文输入框文案
   - 正文图片
   - 可选发布

该 skill 当前聚焦在一个较小且实用的范围：

- 输入来源：播客或视频播客
- 输出目标：小红书图文笔记
- 重点交付物：正文图片

## skill 与 mm-harness 的关系

`podcast-to-xhs-post` 不重复实现长内容理解。

它应遵循如下协作方式：

1. 如果用户提供的是原始 URL、音频、视频或字幕，先调用 `mm-harness create`
2. 如果用户已经提供现成的 harness workspace，则直接消费该目录
3. skill 的创作素材来源优先是 harness workspace 中的结构化结果，而不是重新回到原始媒体逐段处理

## 内容产物

一篇完整的小红书播客笔记由三部分组成：

1. 标题
2. 正文输入框文案
3. 正文图片

其中真正承载详细信息的主要是正文图片。

## 笔记图片设计原则

播客转小红书笔记的图片，不应做成信息图海报，也不应做成一页只放几个 bullet 的卡片。

更合适的形式是：

- 一个视觉吸引力足够的首图
- 一篇正常撰写的正文
- 正文以图片形式分页展示
- 每页适合手机阅读
- 整体像“图文化文章”，而不是“卡片化摘要”

参考样例表明，这类正文图片通常具备这些特点：

- 白底或极浅底色
- 深色正文
- 大字号、较宽行距、较多留白
- 小标题明显
- 正文以自然段为主
- 会适当引用嘉宾或主持人的原话
- 可选插入少量高质量配图，但并非每页都必须有图

## 笔记内容写作规则

笔记内容应基于 harness workspace 提炼而来。

优先提取：

- 播客主题与核心观点
- 值得传播的关键判断
- 可直接引用的嘉宾或主持人原话
- 必要的背景、例子与解释

写作时遵循以下要求：

- 正文按文章方式连续撰写，而不是按单张图逐页独立构思
- 使用小标题组织结构
- 以自然段为主，而不是堆砌关键词
- 可适度引用原话提升可信度
- 重点句可以用粗体、斜体等方式强调
- 正文整体长度应足以支持最终渲染出约 8 到 12 张图片

## Markdown 输入约定

图片渲染输入应使用一个 Markdown 文件。

该文件分成两个部分：

1. 首图页
2. 正文主体

两者之间使用一个 `---` 分隔。

示例：

```markdown
---
title: "笔记标题"
source: "播客来源"
intro_image: "./selected-frame.jpg"
---

# 封面页

这期播客主要讨论了……
我整理出了最值得分享的几个重点。

---

# 第一个关键判断

正文……

## 为什么这很重要

正文……

> 嘉宾原话……

正文……
```

解释：

- YAML frontmatter 用于全局元数据，例如标题、来源和首图路径
- YAML frontmatter 不是正文内容
- YAML frontmatter 之后先写首图页文字内容
- 首图页之后用一个 `---` 分隔正文主体
- frontmatter 中应有 `intro_image`
- 正文主体内部不再通过 `---` 人工分页
- 正文分页由渲染器自动完成

## 渲染器目标

`podcast-to-xhs-post` 使用自己的专用渲染器 `render_xhs.py`。

该渲染器应是一个“阅读型长文分页渲染器”，而不是“多主题卡片渲染器”。

输入：

- 一个 Markdown 文件
- 包含首图页与正文主体

输出：

- `intro.png`
- 自动分页后的正文图片：
  - `page_1.png`
  - `page_2.png`
  - `page_3.png`
  - ...

## 新渲染器的核心行为

### 1. 解析输入

- 读取 frontmatter
- 识别 frontmatter 之后的第一个 `---`
- 将其前面的内容视为首图页
- 将其后面的内容视为正文主体

### 2. 首图页单独渲染

首图页具有独立版式：

- 默认优先使用真实视频播客画面
- 如果没有合适的视频画面，则回退到播客 thumbnail
- 如果 thumbnail 也不合适，则使用与主题相关的外部图片
- 负责展示笔记标题、首图，以及针对该播客的简要介绍
- 不参与正文自动分页

### 3. 正文主体自动分页

正文主体先转成 HTML，再由浏览器排版并自动分页。

分页规则：

- 基于固定页面宽高进行自动切分
- 尽量以块级元素为边界切页
- 不在普通段落中间随意截断
- 标题、引用、图片、列表应尽量完整保留在同一页
- 如果单个块过长，允许在更细粒度处切分，但应尽量避免破坏阅读连续性

### 4. 输出风格

新渲染器不再强调多主题、多风格皮肤。

第一版应采用固定、克制、适合播客笔记阅读的版式：

- 白底或极浅底色
- 深色正文
- 明确但不夸张的小标题层级
- 大字号
- 较宽行距
- 足够留白
- 低存在感页码

### 5. 推荐页数

目标不是压缩成少量卡片，而是形成适合小红书阅读节奏的长图文正文。

推荐正文页数：

- 通常为 8 到 12 页

必要时可以更多，但不应为了追求页数而人为拆碎内容。

## 实现影响

本模块后续实现至少会涉及：

- `src/video_atlas/skills/podcast-to-xhs-post/SKILL.md`
- `src/video_atlas/skills/podcast-to-xhs-post/scripts/render_xhs.py`
- `src/video_atlas/skills/podcast-to-xhs-post/references/params.md`

## 当前结论

`podcast-to-xhs-post` 的正文图片生成应走这条路线：

- 先写一篇完整正文
- 首图页和正文主体只用一个 `---` 分隔
- 首图默认优先使用真实视频播客画面
- 正文主体由专用阅读型渲染器自动分页
- 输出结果应接近真实播客转小红书笔记样例，而不是通用信息卡片
