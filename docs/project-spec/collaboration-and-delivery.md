# 协作与交付规范

## 协作与交付的目标

- 协作与交付的目标，是确保个人工作能够被他人理解、验证、接收和继续维护。
- 交付不仅意味着“代码已经写完”，还意味着相关上下文、风险、验证结果和必要文档已经一并准备完成。
- 协作规范应降低沟通成本、评审成本和后续接手成本，而不是增加不必要的流程负担。
- 任何交付物都应尽量做到边界清晰、说明充分、状态可追踪。

## Git 工作流规范

- 所有改动都应在可追踪的版本控制流程下进行，避免脱离版本管理的临时交付。
- 不同任务的改动应尽量隔离，避免在同一工作流中混入无关变化。
- 对主线有影响的工作，应尽量通过清晰分支或等价隔离方式完成，而不是直接在主线上累积实验性修改。
- 在开始新任务前，应先确认当前工作区状态清晰可解释，避免旧改动与新任务相互污染。

## 提交粒度规范

- 提交应保持小而清晰，每次提交只表达一个相对完整、可解释的变化。
- 不应把多个无直接关系的改动混在同一提交中。
- 结构性重构、行为变更、文档更新、测试补充若影响性质明显不同，应优先考虑拆分提交。
- 一个提交应尽量让 reviewer 和后续维护者能够直接回答：“这次改动到底做了什么”。

## 提交说明规范

- 提交说明应准确表达改动意图，而不是只描述表面动作。
- 提交信息应优先说明行为变化或设计变化，而不是仅列出文件操作。
- 对于重要改动，应确保从提交说明中可以快速判断其主题和影响方向。
- 模糊、过宽或无法体现改动核心的提交说明应尽量避免。

## 变更说明规范

- 对外可见行为变化、公共契约变化、运行方式变化、配置方式变化等，应提供清晰说明。
- 变更说明应尽量回答：
  - 改了什么
  - 为什么改
  - 影响了什么
  - 如何验证
- 若某项改动存在限制条件、兼容性问题或已知风险，应在交付时明确说明，而不是依赖口头记忆。
- 不能假设接收方天然知道该改动的背景和上下文。

## 文档同步规范

- 任何影响公共行为、接口形式、运行方式、目录结构或使用方式的改动，都应同步更新文档。
- 文档更新应视为交付的一部分，而不是后续可选补充。
- 若某项改动需要依赖额外背景才能理解，应尽量把背景写入对应文档，而不是只保留在聊天记录或个人记忆中。
- 已失效、已过时或与实现不一致的文档应及时修正，而不是长期保留。

## 交付前清理规范

- 交付前应清理与当前任务无关的临时修改、实验代码和调试残留。
- 不应把无关文件变更、临时脚本、调试输出、硬编码路径或失效注释带入正式交付。
- 若工作区中存在其他尚未处理的改动，应明确区分哪些属于当前交付、哪些不属于。
- 交付物应尽量保持“接收方打开后即可理解当前变更范围”的状态。

## Review 协作规范

- 提交给他人评审的内容应尽量可读、可解释、可验证。
- 评审协作应优先降低 reviewer 理解成本，而不是把结构复杂性直接转嫁给 reviewer。
- 若改动较大，应主动提供必要上下文、影响说明和验证结果。
- 对评审意见，应以澄清问题、提升质量为目标，而不是仅追求流程完成。

## 风险与限制说明规范

- 若交付内容存在已知限制、临时折中或未彻底解决的问题，应明确说明。
- 不应将高风险、不确定行为或未验证假设隐含带入主线。
- 若某项风险需要后续继续处理，应尽量留下可追踪记录，而不是仅停留在口头约定。
- 对不能立即解决的问题，至少应说明现状、影响和边界条件。

## 安全与边界红线

- 不应提交敏感信息、私有凭证、个人环境配置或不应进入版本控制的本地资产。
- 不应在缺乏说明和验证的情况下交付会破坏既有契约或公共行为的改动。
- 不应通过隐藏问题、弱化说明或省略验证来制造“可交付”的假象。
- 不应把仅在个人环境中成立、无法稳定复现的结果作为正式交付结论。

## 可接手性原则

- 一次好的交付，应保证即使由非作者继续接手，也能理解其目的、边界、状态和后续方向。
- 交付物应尽量让后续维护者能够回答：
  - 这是什么
  - 为什么存在
  - 改动边界在哪里
  - 已验证到什么程度
  - 还有什么限制或后续工作
- 若一项工作只有作者本人能够解释、运行或维护，则说明其交付质量仍不充分。
