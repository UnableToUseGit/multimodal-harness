# 测试规范

## 测试的目标

- 测试的核心目标是验证系统行为是否符合预期，而不仅仅是“代码可以运行”。
- 测试应优先覆盖稳定契约、关键流程和高风险变更点。
- 测试应服务于回归保护、问题定位和后续重构，而不是形式上的覆盖率堆积。
- 新增测试时，应优先保证其能够清晰表达被验证的行为。

## 测试分层原则

- 测试应根据验证目标分层组织，避免所有验证都堆积在单一大流程测试中。
- 单元测试用于验证小范围、单职责、可独立推理的逻辑。
- 集成测试用于验证多个模块协作后的行为是否正确。
- 回归测试用于固定已经发生过问题、已经修复或高风险易回退的行为。
- 对复杂 workflow，应优先同时具备：
  - 关键模块的单元测试
  - 主流程的集成测试
  - 已知风险点的回归测试

## 必须优先覆盖的内容

- 稳定的数据模型与接口契约。
- 输入解析、输出转换和关键格式处理逻辑。
- 公共能力模块的核心行为。
- 持久化格式、目录结构或外部表示形式的稳定约定。
- 主流程中的关键行为路径。
- 配置加载、运行入口和关键参数处理逻辑。
- 已修复 bug 对应的行为边界。

## 单元测试规范

- 单元测试应尽量聚焦单一行为，不同时验证多个不相关目标。
- 单元测试应优先覆盖纯函数、解析逻辑、格式转换、命名规则、公共 helper 和职责清晰的独立模块。
- 单元测试失败时，应能快速定位到具体模块或具体行为。
- 如果某段逻辑可以直接单测，就不要只通过大流程间接验证它。

## 集成测试规范

- 集成测试应验证多个模块协作后的真实行为。
- 主流程级测试应优先验证：
  - 输入是否被正确处理
  - 中间阶段是否被正确串联
  - 最终产物是否符合契约
- 集成测试应尽量避免依赖不稳定的外部环境；若必须依赖，应清楚标注前提条件。
- 能通过 stub、mock 或固定输入隔离不稳定外部依赖时，应优先隔离，以保持测试稳定。

## 回归测试规范

- 任何已确认的 bug、误解析、契约漂移、目录结构问题、路径处理问题，只要被修复，就应尽量补回归测试。
- 回归测试应优先固定“过去出错的最小条件”，而不是顺手扩大成大而模糊的场景。
- 回归测试的命名应能够反映它所防止的具体问题。

## 测试组织规范

- 测试文件统一放在 `tests/` 目录下。
- 测试文件命名使用 `test_*.py`。
- 测试类、测试函数命名应直接表达被验证的行为，避免含义模糊的名称。
- 新增测试时，应优先放在与被测模块语义最接近的位置，保持测试结构清晰可追踪。

## 测试设计原则

- 测试应优先验证行为和契约，而不是过度绑定实现细节。
- 测试应尽量使用最小、明确、可重复的输入。
- 测试失败原因应尽量单一、可读、可定位。
- 避免为了“提高覆盖率”而编写低价值、难维护或无法说明真实行为的测试。
- 当接口、目录结构、外部表示或流程契约发生变化时，应同步更新测试，而不是保留已失效的旧预期。

## 提交前的最小验证要求

- 对新增功能或行为变更，至少应运行与改动直接相关的测试。
- 对影响核心流程、公共能力或公共契约的改动，应运行完整测试集。
- 提交前至少应确认：
  - 新增测试通过
  - 相关旧测试未被破坏
  - 没有因临时调试改动导致测试结果失真

## 测试与实现的一致性要求

- 实现变更后，测试和文档应同步更新，避免三者之间出现长期漂移。
- 若某项行为已经没有测试保护，应视为高风险区域。
- 若测试难以编写，通常说明模块边界、职责划分或输入输出设计仍需改进。
- 测试不是实现完成后的补充材料，而是行为契约的一部分。
