# 2026-04-01: Canonical Atlas 输入分层与职责划分

## 背景

随着 CLI 和 URL acquisition 能力引入，当前实现已经不再只是单纯的业务逻辑层。系统中实际出现了三类不同性质的职责：

- 面向用户或 Agent 的输入接收与参数解析
- 将外部输入整理为业务层可消费输入的适配与编排
- canonical atlas 的核心业务执行

如果不尽快明确这些职责边界，后续继续支持 URL、本地文件、视频、音频、字幕、多来源 metadata 时，CLI、workflow 和 acquisition 逻辑会进一步耦合，难以演进。

本文档用于记录当前阶段达成的一致设计结论。

## 决策摘要

系统采用分层架构，至少划分为以下三层：

- `Interface Layer`
- `Application Layer`
- `Business Layer`

其中：

- `Interface Layer` 负责接收用户输入并完成最薄的一层参数解析
- `Application Layer` 负责将用户输入整理、落盘并转换为业务层可消费的标准请求
- `Business Layer` 负责 canonical atlas 的实际业务处理

`transcription` 的具体实现不是 interface 或 application 的职责；是否触发转写应由业务层根据输入条件决定。

## 分层定义

## Interface Layer

### 目标

提供面向用户或 Agent 的交互入口，例如 CLI。

### 输入

- 命令行参数
- 未来可能扩展为 Agent skill 调用参数或 API 请求参数

### 输出

- 解析后的结构化参数

### 职责

- 接收用户输入
- 进行基础参数解析和必要校验
- 将解析结果传递给 application layer

### 非职责

- 不直接执行下载、复制、转写、planner、parsing、compose
- 不直接决定业务处理路径

## Application Layer

### 目标

将 interface layer 传入的结构化输入，转换为业务层能够稳定消费的标准业务请求，并完成输入资产的准备与落盘。

### 输入

- interface layer 解析后的参数

### 输出

- `CanonicalCreateRequest`
- `CanonicalPipelineConfig`

### 核心职责

1. 识别输入来源与输入类型
   - URL 输入
   - 本地文件输入
   - 视频、音频、字幕及附加 metadata
2. 对外部输入进行 acquisition 或 materialization
   - 对 URL 执行下载、metadata 抓取、字幕抓取等操作
   - 对本地文件执行 copy 或标准化落盘
3. 将输入资产统一整理到目标目录下的子目录中
   - 采用 `uid` 命名的子目录，避免不同输入相互覆盖
4. 组装标准业务请求
   - 生成 `CanonicalCreateRequest`
   - 加载并传递 `CanonicalPipelineConfig`
5. 调用 business layer

### 当前目标输入范围

当前讨论中，用户输入主要包括两类：

1. 提供 URL
   - 视频 URL
   - 播客音频 URL
2. 提供本地文件
   - 视频文件
   - 音频文件
   - 字幕文件
   - 可选 metadata

### URL 输入的当前策略方向

对于 URL 输入，application layer 负责：

- 获取可用 metadata
- 根据来源类型和策略决定需要抓取哪些文件
- 将抓取结果直接落到目标目录下的 `uid` 子目录中

当前已明确的策略方向包括：

- 视频 URL 支持基于 metadata 决定 acquisition 行为
- 音频 URL 直接抓取音频文件
- 小宇宙这类播客 URL 可以优先采用轻量页面解析方式提取音频直链与基础 metadata，再下载音频文件

其中类似“视频长度超过 25 分钟时仅抓取字幕，否则同时抓取字幕和视频”的规则，应视为 application layer 中的 acquisition policy，而不是 interface layer 的行为。

说明：

- 这部分是当前讨论形成的目标架构方向，不代表仓库现阶段已经支持所有来源
- 当前已落地的 URL acquisition 仍以 YouTube 为主
- 小宇宙音频下载的可行性已通过参考脚本(docs/references/Agent-Reach/agent_reach/scripts/test.sh)验证，后续实现时可新增独立 source-specific acquisition backend，而不应把页面解析逻辑直接放入 CLI 或 workflow
- 当前实现中，YouTube acquisition 已采用“先抓 metadata，再根据时长决定是否下载视频”的两阶段策略；默认阈值为 `25` 分钟

### 非职责

- 不直接执行 canonical atlas 的核心业务逻辑
- 不负责决定业务层是否需要转写
- 不负责 planner、parsing、compose 的具体处理

## Business Layer

### 目标

根据 application layer 提供的标准业务请求与配置，执行 canonical atlas 的核心业务处理。

### 输入

- `CanonicalCreateRequest`

### 输出

- `CanonicalAtlas`

### 职责

- 根据 request 判断当前业务路径
- 在必要时执行转写
- 执行 planner、parsing、compose 等核心流程
- 生成 canonical atlas 结果

### 建议入口形式

business layer 的正式接口应演进为：

```python
create(request) -> CanonicalAtlas
```

其中：

- `request` 表达本次处理的输入内容与已准备好的输入资产
- workflow 所需的模型配置与运行参数应在实例化阶段完成绑定，而不是继续作为 `create(...)` 的离散运行参数传入

application layer 负责加载 `CanonicalPipelineConfig` 并实例化 workflow；business layer 的单次执行入口只消费 `request`。

### 业务路径

当前已明确的 canonical atlas 业务路径包括：

1. 纯文本流程
   - 适用于小宇宙 URL、本地音频文件、本地字幕文件，以及只抓取到字幕的长 YouTube 视频
   - 若输入是音频，则先转写成字幕
   - 若输入是字幕，则直接利用
   - 后续统一执行 text-based `planner -> parsing -> compose`
2. 视频+文本流程
   - 适用于本地视频文件，以及可抓取视频文件的 YouTube 视频
   - 视频是主输入
   - 字幕是强辅助输入
   - 若用户已提供字幕，则直接利用
   - 若用户未提供字幕，则应尽力补齐，优先复用已有字幕，否则抽音频并转写

### 业务路径判定规则

business layer 应按“可用业务资产”而不是“输入来源类型”判断流程：

- 若 request 中存在可用视频资产，则进入视频+文本流程
- 若 request 中不存在视频资产，但存在可用字幕资产或可转写音频资产，则进入纯文本流程
- 若以上条件都不满足，则直接报错

补充说明：

- 视频 metadata 和音频 metadata 都是可选输入
- 若 metadata 可提供，则应允许业务层消费，以提升处理质量
- 业务层关注的是当前 request 能否提供视频、字幕或可转写音频，而不是这些资产最初来自 URL 还是本地文件

## 关于 Transcription 的职责归属

结论：

- `transcription` 能力本身不属于 interface layer 或 application layer
- 是否触发转写，属于 business layer 的职责

原因：

- 转写并不是单纯的输入适配，而是业务执行路径的一部分
- 它会影响后续 planner、parsing 和 compose 的实际行为
- 不同输入条件下是否需要转写，应由业务层统一判断，而不是在 application layer 提前隐式执行

因此：

- application layer 可以准备音频、视频、字幕等输入资产
- business layer 决定是否需要调用 transcriber

## 关于源文件复制与下载职责的结论

结论：

- 将 source video、source audio、source subtitles 下载或复制到目标目录的职责，属于 application layer
- canonical workflow 不应再承担“把源文件复制到指定目录”这一职责

原因：

- 无论输入来自 URL 还是本地文件，这本质上都属于输入准备与 materialization
- 这类文件搬运行为不应混入 canonical atlas 的核心业务流程
- 业务层应尽量消费已经准备好的标准输入资产，而不是自行处理来源文件布局

## 对后续实现的影响

后续代码结构应逐步向以下方向演进：

- `CLI` 保持在 interface layer
- 新增明确的 application layer 模块，负责输入适配、acquisition、materialization 和 request 组装
- canonical workflow 保持为 business layer，专注于 canonical atlas 业务执行
- source acquisition 作为 application layer 依赖的能力模块存在

同时，应逐步将以下职责从 workflow 中迁出：

- 源文件复制到目标目录
- URL 输入准备
- 与用户输入形式直接相关的分发逻辑

## 待后续细化的问题

以下问题已形成方向，但尚未在代码中完全定型：

- `CanonicalCreateRequest` 的正式字段设计
- application layer 的模块命名与目录位置
- 音频 URL 与本地音频输入的统一建模
- metadata 在 request 中的精确表示方式
- application policy 与 business policy 的进一步边界
