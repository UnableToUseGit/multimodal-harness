# 2026-04-02 Planner 与 Profile 简化决策

## 背景

在 `text-first canonical workflow` 的推进过程中，当前实现暴露出一个结构问题：

- `plan` 被简化成了 route gate
- 但在原始 canonical workflow 中，`plan` 的真正职责是：
  - 根据输入上下文判断内容 `profile`
  - 生成全局语义描述
  - 为后续 parsing 构造 execution plan

这说明新 workflow 不能只保留一个轻量 route selector，而应恢复为更完整的 planning 职责。

## 核心结论

### 1. 统一只保留一个 planner

不再按输入来源拆成多个 planner。

统一由一个 planner 处理以下输入情况：

- 有视频且可提供视觉采样
- 有视频但当前只能提供字幕
- 音频

planner 接收尽量完整的上下文后，统一判断内容 `profile` 并生成全局描述。

### 2. planner 的职责

planner 的职责收敛为两件事：

- 判断当前数据的 `profile`
- 生成针对当前数据的 `concise_description`

`planner` 不直接输出最终 router，也不直接输出最终 execution plan。

### 3. planner 输入

planner 输入由 business layer 在进入 planning 前统一整理。

输入包括：

- 被处理数据类型
  - 视频且可提供视觉采样
  - 视频但当前只能提供字幕
  - 音频
- 数据采样
  - 对于可提供视觉采样的视频：稀疏采样帧 + 部分字幕
  - 对于视频但只能提供字幕、音频：部分字幕
- `source_metadata`
  - 若 application layer 提供，则作为可选增强上下文传入 planner

最终这些信息会被统一整理成 prompt，作为 planner 的输入。

### 4. planner 输出

planner 输出收敛为：

- `genres`
- `profile`
- `concise_description`

### 5. `profile` 不是 `segmentation_profile`

planner 判断的是内容 `profile`，而不是更低层的 `segmentation_profile`。

原因是：

- `profile` 表示内容类型判断
- `route`、`segmentation_policy`、`caption_policy` 都是从 `profile` 派生出来的执行策略

因此，planner 输出层应停留在 `profile` 这一层，不应直接下沉到更具体的 parsing 配置。

## Profile 设计调整

### 1. 废弃当前拆分式 profile 设计

当前 `canonical_registry.py` 中分别定义：

- `SegmentationProfile`
- `CaptionProfile`

这套设计在当前阶段显得过重，也不利于 text-first workflow 的演进。

### 2. 引入统一 `Profile`

后续建议将 profile 简化为统一结构：

```python
@dataclass(frozen=True)
class Profile:
    route: str
    segmentation_policy: str
    caption_policy: str
```

当前明确不保留以下字段：

- `target_segment_length_sec`
- `signal_priority`

这两个字段在现阶段先直接废弃，不再作为 profile 的一部分。

### 3. registry 的职责

registry 的职责应调整为：

- 维护 `profile_name -> Profile` 的映射

也就是说：

- planner 输出 `profile`
- 程序根据 `profile` 从 registry 中解析出统一 `Profile`
- 再由该 `Profile` 进一步指导 parsing / captioning

### 4. 第一版统一 profile 集合

第一版统一 profile 名收敛为：

- `lecture`
- `podcast`
- `explanatory_commentary`
- `movie`
- `drama`
- `sports`
- `other`

说明：

- 原先更细粒度的 `interview` 暂时并入 `podcast`
- 原先较模糊的 `knowledge` 改为 `explanatory_commentary`

### 5. 第一版 route 映射

第一版 route 映射如下：

- `lecture` -> `text_first`
- `podcast` -> `text_first`
- `explanatory_commentary` -> `text_first`
- `movie` -> `multimodal`
- `drama` -> `multimodal`
- `sports` -> `multimodal`
- `other` -> `multimodal`

其中：

- `other` 默认映射到 `multimodal`
- 这样可以避免将不确定内容静默降级到 `text_first`
- 在当前 v1 阶段，`multimodal` 仍然意味着显式返回不支持

## Execution Plan 的职责

`execution plan` 机制不应被废弃，只应被简化。

后续结构应是：

- planner 输出：
  - `genres`
  - `profile`
  - `concise_description`
- 程序 resolve `profile`
- 程序基于 resolved `Profile` 与 planner 输出构造 execution plan

因此：

- planner 不直接产出 execution plan
- business layer 仍然负责将 planner 输出转成真正可执行的 plan

## 对 text-first workflow 的含义

这项决策意味着：

- `text-first canonical workflow` 中的 `plan` 不应只是 route gate
- 它应恢复成真正的 planning 阶段
- parsing 不应再硬编码执行，而应消费 planning 结果

同时，当前 route 判断逻辑仍保留：

- 若 resolved `Profile.route == "text_first"`，继续执行
- 若 resolved `Profile.route == "multimodal"`，在 v1 阶段显式返回不支持

## 当前状态说明

本决策文档描述的是后续重构方向，不代表仓库当前实现已经完全满足上述设计。

当前仓库中的 text-first workflow 仍处于过渡阶段，后续应继续按本决策进行整理。
