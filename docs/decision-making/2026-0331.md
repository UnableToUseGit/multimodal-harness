# 2026-03-31 Canonical Atlas 两阶段结构思考

## 背景

在实现 `text_llm` 路线并跑通一轮验证后，我重新审视了 canonical atlas 当前的生成流程，重点关注以下两个问题：

1. 文本叙事类视频虽然已经可以通过 text route 更高效地完成切分，但生成结果仍然难以稳定达到我期望的“最终结构”。
2. 目前 canonical atlas 默认把 “boundary detection” 和 “最终结构形成” 视为同一件事，这可能是问题的根源。

## 当前观察

### 1. 为了方便验证，临时要求 text route 输出中文

- 当前验证依然比较费劲，因为模型输出通常是英文，而视频内容很多是中文。
- 为了降低评测成本，我临时在 `TEXT_BOUNDARY_DETECTION_PROMPT` 中加入了“字段内容尽量用中文填写”的要求。
- 这是一项验证期优化，不应直接视为最终长期设计结论。

### 2. title 生成职责已开始前移

- `title` 的生成职责正在由 assemble 模块转移到 `video_parsing.py`。
- 目前 text 路线的相关逻辑已完成调整。
- multimodal 路线的对应改造尚未完成。

这一变化本身说明：随着流程职责逐步细化，`video_parsing` 不再只是“检测边界”，而开始承担更明确的 unit 级语义产出职责。

## 问题定义

我发现，无论如何调整 `segmentation_policy`，都很难让 boundary detection 直接产出我期望的“最终结构”。

更具体地说：

- 我准备的 case 中有一个来自哔哩哔哩的 podcast 视频。
- 该视频时长约 2.5 小时，视频作者自己已经划分了六七个较大的章节，供观众观看。
- 即便我在 `segmentation_policy` 中明确要求：
  - 不要按 `sub-topic` 切
  - 应将完整的 `discussion` 作为一个 segment
- 模型仍然倾向于切得很碎。

## 根因判断

我认为问题不在于模型完全不理解这些词，而在于：

- `sub-topic`
- `topic`
- `discussion`

这些概念本身是层级化的、模糊的、缺乏稳定操作定义的。

对于 boundary detection 而言，它更适合回答的问题是：

- 这里是否形成了一个局部语义边界？
- 这里是否已经可以构成一个稳定的基本单元？

而它不擅长直接回答：

- 这几个连续片段是否属于同一个更大的 discussion block？
- 最终 atlas 中应该保留哪一层结构？

因此，把“最终结构生成”完全压在 boundary detection 上，是一种职责错配。

## 决策

`Video -> Atlas` 应拆分为两个阶段：

### Stage 1: Boundary Detection / Unit Detection

- 目标：把视频切分成稳定的基本单元（units）
- 输入：视频、字幕、切分策略
- 输出：一组顺序稳定、局部语义相对完整的 `units`
- 特点：
  - 可以比最终结构更细
  - 重点是局部边界判断
  - 不要求直接等于最终 atlas 结构

### Stage 2: Structure Composition

- 目标：在 `units` 基础上，按照更高层的结构要求组合成最终 `segments`
- 输入：全体 `units` 的文本信息
- 输入：用户可选提供的结构要求
- 输出：最终 atlas 的 `segments`
- 特点：
  - 不再直接读取视频内容
  - 主要做结构组织，而不是重新做视频理解
  - 更适合表达 “discussion / chapter / phase / plot block” 这类上层概念

## 用户结构要求

考虑到用户本身可能对最终结构有明确要求，canonical atlas 的对外接口应允许显式承接这类输入。

因此，未来在 `CanonicalAtlasWorkflow.create(...)` 中应增加一个可选参数：

- `structure_request: str | None = None`

该参数用于表达用户对最终 atlas 结构的要求，例如：

- “帮我把这个视频分成 4-6 部分”
- “帮我按较粗的章节划分”
- “尽量保持较大的 discussion block”
- “按比赛的大阶段组织”

### 边界约束

- `structure_request` 只供 Stage 2 使用
- Stage 1 不直接消费 `structure_request`
- Stage 1 仍按 profile、route 和 segmentation policy 切出稳定 `units`
- Stage 2 在读取 `units` 文本信息的基础上，再结合 `structure_request` 组织最终 `segments`

这样可以避免把“最终结构要求”重新压回 boundary detection 阶段。

## 为什么 Stage 2 需要 LLM

我倾向于让 Stage 2 由 LLM 完成，而不是单纯规则合并。

原因：

- 最终结构不是纯长度阈值或关键词规则能稳定表达的。
- 不同视频类型的高层结构差异很大：
  - podcast 里的 discussion block
  - lecture 里的章节
  - documentary 里的故事段落
  - match broadcast 里的 phase block
- 如果 Stage 2 只消费 `units` 的文本表示，它的推理成本会明显低于重新看视频。

因此，当前更合理的方向是：

- Stage 1 用 `segmentor` 切 `units`
- Stage 2 用独立的 LLM composer 组合 `units` 成 `segments`

## Stage 2 的输入与输出

### 输入

Stage 2 暂定只读取 `units` 的文本信息，而不重新读取视频内容。

此外，Stage 2 应额外接收：

- `structure_request`

每个 unit 至少应提供：

- `unit_id`
- `start_time`
- `end_time`
- `duration`
- `title`
- `summary`
- `caption`
- `subtitles_text`

可选附带：

- `boundary_rationale`

### 输出

Stage 2 不应只输出自由文本，而应输出可落盘的半结构化结果。

每个最终 segment 至少应包含：

- `segment_id`
- `unit_ids`
- `title`
- `summary`
- `composition_rationale`

## 当前实验期目录结构设想

为了方便验证与调试，我希望实验期 atlas 目录同时保留两层结果：

```text
atlas_dir/
  README.md
  units/
    unit_0001/
      README.md
      video_clip.mp4
      SUBTITLES.md
    unit_0002/
      ...
  segments/
    seg_0001/
      README.md
      unit_0001/
        README.md
        video_clip.mp4
        SUBTITLES.md
      unit_0002/
        README.md
        video_clip.mp4
        SUBTITLES.md
    seg_0002/
      ...
```

### 说明

- `units/`：保留 Stage 1 的原始输出，主要服务调试与分析
- `segments/`：作为当前人工 review 的主入口
- `segments/<seg>/unit_x/`：实验期允许复制完整媒体文件，目的是方便验证 composer 的组合是否合理

这种结构存在冗余，但当前阶段以验证效率为优先。

长期来看，我的目标仍然是：

- 最终正式版本只保留 `segments/`
- `units/` 仅作为实验与调试产物，未来可移除

## 当前范围内不做的事

本次 decision 只确立两阶段结构方向，不在当前阶段直接解决以下问题：

- 超长视频下，Stage 2 如何分块组合
- Stage 2 如何做增量组合或滑动窗口组合
- `units/` 与 `segments/` 的最终正式落盘契约
- multimodal 路线 title 生成职责前移的完整收口
- review app 如何同时高效浏览 units 和 segments

## 结论

当前 canonical atlas 的主要问题不是 `segmentation_policy` 写得不够好，而是流程职责划分存在错位。

更合理的方向是：

1. Stage 1 只负责切出稳定 `units`
2. Stage 2 基于 `units` 文本信息和用户可选提供的 `structure_request` 生成最终 `segments`
3. 当前实验期 atlas 目录同时保留 `units/` 与 `segments/`
4. `segments/` 作为主 review 入口，允许复制完整 unit 媒体文件以支持调试

这条路线比继续强迫 boundary detection 直接产出最终结构更有希望得到稳定结果。
