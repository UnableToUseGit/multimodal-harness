# Derived Atlas Agent

## Purpose: 

Given a user task description and a canonical atlas, this module derivates a newly task-oriented atlas. This atlas provide necessary information required by tasks.

## Responsibilities: 

- Accept task request and analyze what information does it requrie?
- Select `AtlasSegment` from `CanonicalAtlas` contained the required information and process them into `DerivedAtlas`.
- Write the `DerivedAtlas` into `workspace`.

## Inputs:

- `task_request` (`str`, required): a user request that describs what he want to do with a video.
- `canonical_atlas` (`CanonicalAtlas`, required): a domain model represent the video cannonical atals generated by Canonical Atlas Agent.
- `workspace` (`BaseWorkspace`, required): the workspace used for contain generated video atlas.
- `planner` (`BaseGenerator`, required): component used to providing instruction for `segmentor`.
- `segmentor` (`BaseGenerator`, required): component used for boundary detection
- `captioner` (`BaseGenerator`, required): component used for caption or other text information generation.
- `num_workers` (`int`, optional): the number of workers participated in derivation process.

## Outputs:

- `DerivedAtlas` (`object`): a domain model represents an atlas derived from canonical atlas given the `task_request`.
- `DerivationResultInfo` (`object`): a domain model contains statistics or summary of derivation.
- A file directory (`workspace`) correpsonding to `DerivedAtlas`

## Dependencies:

- `CanonicalAtlasAgent`: Its output `CanonicalAtlas` is the target for `DerivedAtlasAgent` to process.
- `Generator`: A module that provides a lightweight interface for LLM interactions. The `planner`, `segmentor`, and `captioner` components are derived from this module.
- `Workspace`: A module that provides a bridge between code and os/file system. It can executes the command predefined or generated to operate file system.

## Internal Workflow:

1. **CandidateGeneration**

This module firstly use `planner` to read all `AtlasSegment` and `task_request` to select those that contain task-required information. And generate a derivation policy `DerivationPolicy` for each one.

2. **Derivation**

Then this module will derive task-aware atlas from each selected `AtlasSegment` following its assigned `DerivationPolicy`. Derivation process for each `AtlasSegment` contains three steps: 
- Re-grounding: Raw video clips retrieved from the `CanonicalAtlas` are typically coarse-grained. Therefore, it is often necessary to localize fine-grained sub-clips that precisely align with the task requirements.
- Subtitle Pruning: Retain only the subtitles associated with the re-grounded sub-clips.
- Re-caption: generate title, summary, detailed caption for these fine-grained sub-clips.

Notably, derivation process for each `AtlasSegment` should be executed in parallel.

Key derivation details should be encapsulated in `DerivationResultInfo` for subsequent analysis. These include the count of derived AtlasSegments, the `DerivationPolicy` for each segment, and the source `AtlasSegment` from which each was derived.

Critical Output:
- A list of derived `AtlasSegment`
- `DerivationResultInfo` 

3. **Aggregation**

This step aggregates all derived `AtlasSegment`s and their corresponding derivation policies to generate a comprehensive **derivation conclusion**. The conclusion comprises two key components:
- Global Summary: Provides aggregate statistics, including the total count of derived segments, their cumulative duration, and the average segment duration.
- Detailed Breakdown: Specifies the contribution of each derived AtlasSegment to the task. For every segment, it lists the specific information provided, along with its precise temporal boundaries (e.g., start/end timestamps in ISO 8601 strings).

Finally, both the generated conclusion and the list of derived `AtlasSegment`s are encapsulated into the `DerivedAtlas` object.

Critical Output:
- `DerivedAtlas`

4. **Workspace Finalization**
Write `DerivedAtlas` in `workspace` and write `DerivationResultInfo` in its sub-directory `.agentignore`.

## Constraints / Notes:
- Ensuring efficiency, parallel is necessary when `Derivation`
- Derivation policy for each segment should indicate what information is reuqired in this segment. This is important for `Re-grounding`.
