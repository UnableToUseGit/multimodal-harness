# parsing 模块设计

## 文档目标

本文档用于说明 `parsing` 模块的设计目标、职责边界、核心接口、依赖关系和关键流程。本文档面向开发者，目标是帮助读者快速理解该模块在系统中的位置以及它应如何被使用和维护。

## 模块概览

- 名称：`parsing`
- 路径：`src/video_atlas/parsing`
- 主要职责：对模型输出进行清洗、提取和结构化解析

## 职责与边界

### 职责

- 负责对模型返回的文本进行基础清洗。
- 负责从输出中提取结构化内容。
- 为上层 workflow 提供稳定的解析能力。

### 不负责的内容

- 不负责模型请求发送。
- 不负责业务语义决策。
- 不负责下游持久化或展示。

## 核心接口

### `strip_think_blocks`

- 类型：`function`
- 作用：清理模型输出中的非目标推理块或噪声内容。
- 输入：
  - 模型返回的原始文本响应
- 输出：
  - 清洗后的文本
- 说明：
  - 用于减少后续结构化提取时的干扰。

### `extract_json_payload`

- 类型：`function`
- 作用：从复杂文本中提取目标 JSON 载荷。
- 输入：
  - 已清洗或未清洗的模型输出文本
- 输出：
  - JSON 字符串片段
- 说明：
  - 该接口不负责将字符串反序列化为 Python 对象。

### `parse_json_response`

- 类型：`function`
- 作用：将模型输出解析为结构化 JSON 对象。
- 输入：
  - 模型输出文本
- 输出：
  - 解析后的 Python 结构化对象
- 说明：
  - 该接口通常会串联清洗与提取步骤，为上层提供统一解析入口。

## 依赖关系

### 上游依赖

- canonical workflow
- derived workflow
- 其他需要解析模型输出的上层模块

### 下游依赖

- Python 标准库中的字符串处理与 JSON 能力

## 内部组成

### 文本清洗部分

- 角色：处理多余推理内容和无关输出。
- 边界：
  - 负责原始文本的基础清洗
  - 不负责业务级校验和容错策略
- 输入：
  - 模型原始输出文本
- 输出：
  - 清洗后的文本

### 载荷提取部分

- 角色：从复杂文本中提取结构化片段。
- 边界：
  - 负责识别目标 JSON 片段
  - 不负责反序列化后的语义解释
- 输入：
  - 原始或清洗后的文本
- 输出：
  - JSON 字符串载荷

### 结构化解析部分

- 角色：将提取结果转换为 Python 对象。
- 边界：
  - 负责 JSON 反序列化
  - 不负责领域对象构造
- 输入：
  - JSON 字符串载荷
- 输出：
  - Python 结构化对象

## 关键流程

1. 上层模块接收模型原始输出。
2. 模块先执行必要的文本清洗。
3. 模块从文本中提取目标 JSON 载荷。
4. 模块输出解析后的结构化结果供上层消费。

## 设计约束

- 该模块只负责解析，不负责业务验证。
- 该模块应保持对上层 workflow 的独立性。
- 若模型输出格式发生变化，应优先在该模块中集中收敛解析逻辑。
