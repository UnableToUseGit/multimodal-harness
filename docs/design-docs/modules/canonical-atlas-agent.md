# Canonical Atlas Agent

## Purpose:

Build a video canonical atlas from one source video. The canonical atlas is the stable base workspace for later browsing, review, and task-aware derivation. It should reorganize a long video into a structured directory that both humans and downstream agents can navigate reliably.

## Responsibilities:

- parse the full video into stable segments with local captions
- generate global video metadata and final segment titles
- write the these **generated canonical atlas** into workspace (file directory)
- also preserve inspectable intermediate artifacts such as the execution plan and boundary-debug outputs in workspace

## Inputs:

- `planner` (`BaseGenerator`, required): component participates in producing canonical atlast construction plan.
- `segmentor` (`BaseGenerator`, required): component used for boundary detection
- `captioner` (`BaseGenerator`, required): component used for local segment caption and global text information generation.
- `workspace` (`BaseWorkspace`, required): the workspace used for contain generated video atlas.
- `transcriber` (`BaseTranscriber`, optional): component used for audio extraction and subtitls generation.
- `generate_subtitles_if_missing` (`bool`, optional): a flag indicate whether to utilze `transcriber` for subtitles generation.
- `chunk_size_sec` (`int`, optional): the temporal length of video chunk fed into `segmentor` and `captioner`. It's unit is second.
- `chunk_overlap_sec` (`int`, optional): the overlap length between the current video chunk and the last chunk.
- `caption_with_subtitles`  (`BaseWorkspace`, optional): whether to include subtitles informtion when `captioner` is generating caption.
- `input_path` (`str`|`Path`, optional): the file directory you want to processed with video atals. It usually contain a video file and `.src` file.
- `video_path` (`str`|`Path`, optional): the video file path you want to process.
- `subtitle_path` (`str`|`Path`, optional): the subtitle file path you want to process.
- `verbose` (`bool`, optional): whether to print intermediate information.

## Outputs:

- `CanonicalAtlas`: a domain model represent the final generated video cannonical atals.
- A file directory (`workspace`) correpsonding to `CanonicalAtlas`
- A series of intermediate files inlcude `EXECUTION_PLAN.json`, subtitles.srt, audio file `boundary_debug` saved in `workspace/.agentignore/` or `workspace`


## Dependencies:
- `Generator`: the module provides lightweight and convenient LLM calling. 
- `Transcription`: the module provides transciption service (i.e. audio extraction and subtitles generation)
- `Workspace`: the module provide a execution environment for agent. It executes the action generated by agent in file system.

## Internal Workflow:

1. **Plan**
Read probe samples from the source video and optional subtitles. The planner only decides `planner_confidence`, `genre_distribution`, `segmentation_profile`, and `sampling_profile`.

2. **Construct Execution Plan**
Resolve the planner output into a `CanonicalExecutionPlan`. This step expands stable registry defaults such as segmentation profile, caption profile, signal priority, target segment scale, and concrete frame-sampling settings.

3. **Video Parsing**
Process the source video chunk by chunk with overlap. For each chunk:
- detect candidate boundaries inside the core window
- write boundary-debug JSON for inspection
- materialize committed segments
- generate local segment captions asynchronously

This stage is responsible for parsing only. It does not generate final titles or final segment folder names.

4. **Atlas Assembly**
Take all parsed segments together and generate:
- a global video title
- a global abstract
- final canonical segment titles

All of these are packed in `AtlasSegment` and `CanonicalAtlas` and then are writen into final workspace.
 
5. **Workspace Finalization**
Validate the workspace structure and move intermediate artifacts such as `EXECUTION_PLAN.json` into `.agentignore/`.

## Constraints / Notes:

- canonical atlas is not a highlight extractor
- segment coverage should remain continuous and defensible
- planning decisions should stay minimal; execution details are resolved in code
- final segment titles are generated globally, not during parsing
- folder names should only be created after final titles exist
- runtime efficiency matters: probe sampling is lightweight, parsing is chunked, and segment caption generation remains concurrent
- boundary-debug artifacts exist for iteration and prompt tuning, not as public workspace surface
